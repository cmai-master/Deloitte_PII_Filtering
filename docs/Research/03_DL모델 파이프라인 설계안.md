# **0\. 문서 개요**

본 문서는 LG U+ Server-i 플랫폼의 PII 오탐·정탐 분류 모델 구축을 위한 딥러닝 모델 파이프라인의 상세 설계서입니다. 3/19 회의 결정사항과 미결 리서치 항목을 함께 관리합니다.

| 📌 설계 범위 및 전제 • 대상 데이터: 마스킹(Asterisk)된 PII 패턴 텍스트 \+ 앞뒤 \~10자 컨텍스트 • 분류 목표: TP(정탐) 1종 \+ FP(오탐) 7종, 총 8개 클래스 • 학습 데이터: 합성 데이터(Train 80% / Val 20%) → 실데이터(Test holdout) |

# **2\. DL 모델 파이프라인 (Model Pipeline)**

PoC 단계의 최고 성능 모델(TF-IDF, XGBoost 등)을 Baseline으로 초기 지표를 설정하고, 이를 넘어서는 것을 DL 모델의 1차 목표로 삼습니다.

| ModelPipeline ├── 2.1  Encoder │    ├── TextEncoder        \# DeBERTa / KcELECTRA / KLUE-RoBERTa 등 │    └── TabularEncoder     \# FT-Transformer (Tabular 실데이터 확보 후 추가) ├── 2.2  FusionLayer        \# Phase1: pass-through / Phase2: Concatenate ├── 2.3  ClassificationHead \# \[CLS\]→Dropout→Linear→ReLU→Dropout→Linear→Softmax ├── 2.4  LossFunction       \# CrossEntropyLoss → Focal Loss (조건부 전환) └── 2.5  MetricsTracker     \# Macro F1, Per-class F1, Confusion Matrix |
| :---- |

## **2.1  인코더 (Encoder) 아키텍처**

| 📋  \[3/19 회의 결정사항\] • DeBERTa, KcELECTRA 등 개인정보·구어체·노이즈에 특화된 모델들을 우선 검토하되, 학습 데이터 목적성을 논문을 통해 파악할 것. • 아키텍처 스펙 표(헤드 수, 임베딩 레이어 등)를 작성하여 모델 사이즈를 명확히 파악할 것 (액션 아이템). • 서빙 제약 고려: 각 후보 모델별 예상 추론 Latency를 스펙 표에 함께 기재하여 비교. |
| :---- |

**TextEncoder**

Text Encoder 최종 권고 모델은 “mDeBERTa-v3-base”이며, 주요 선정 근거는 아래와 같습니다. 만약 초저지연 시간(10ms 이내)이 필수적인 실시간 환경이라면 BERT-tiny나 DistilBERT를 추천합니다.

1. **다국어 통합 대응:** CC100 다국어 사전학습을 기반으로 한국어·영어·특수문자 혼합 입력을 단일 아키텍처 내에서 효율적으로 처리 가능  
2. **실증적 성능 우위:** PII 탐지 태스크에서 Recall 98% 및 F1 95% 이상의 독보적인 벤치마크 지표를 통해 최고 수준의 성능 입증  
3. **구조적 학습 유리:** Disentangled Attention 메커니즘이 \[MASK\_N\]@domain.co.kr 패턴의 내용과 위치 정보를 분리 학습하는 데 기술적으로 최적화

| 모델명 | 파라미터 수 (Backbone) | 레이어 수 | 헤드 수 | 은닉층 크기 | 임베딩 차원 | 예상 Latency (T4 GPU, Seq 128\) |
| :---- | :---- | :---- | :---- | :---- | :---- | :---- |
| **BERT-base** | 110M | 12 | 12 | 768 | 768 | 3.8ms \~ 5.0ms |
| **DistilBERT** | 66M | 6 | 12 | 768 | 768 | 2.0ms \~ 2.6ms |
| **BERT-tiny** | 4M | 2 | 2 | 128 | 128 | 1.0ms 미만 (GPU) / 10ms (CPU) |
| **DeBERTa-v3-base** | 86M (전체 184M) | 12 | 12 | 768 | 768 | 15ms \~ 25ms |
| **DeBERTa-v3-small** | 44M (전체 143M) | 6 | 12 | 768 | 768 | 10ms \~ 15ms |
| **mDeBERTa-v3-base** | 86M (전체 278M) | 12 | 12 | 768 | 768 | 25ms \~ 33ms |
| **KcELECTRA-base-v2022** | 110M | 12 | 12 | 768 | 768 | 7ms \~ 22ms |
| **KLUE-RoBERTa-base** | 125M | 12 | 12 | 768 | 768 | 6ms \~ 20ms |

**TabularEncoder**

Tabular Encoder 최종 권고 모델은 “FT-Transformer”이며, 주요 선정 근거는 아래와 같습니다. 만약 데이터셋의 크기가 10,000행 미만으로 작고 빠른 배포가 중요하다면 TabPFN v2를 추천합니다.

1. **범용적 성능 우위:** 수치형과 범주형 데이터를 통합적으로 모델링하는 능력이 뛰어나며, 다양한 정형 데이터셋에 대해 안정적인 SOTA 성능을 보장  
2. **고도화된 하이브리드 탐지:** 시맨틱 타입 탐지 시 Sherlock 모델 기반의 하이브리드 방식(Detect-then-Reflect)을 결합하여 오탐지(False Positive)를 최소화하는 데 효과적  
3. **즉각적인 분류 효율성:** 별도의 복잡한 모델 학습 과정 없이도 정형 데이터 내 컬럼 분류 및 PII 식별 과업에서 즉각적이고 높은 정확도 산출 가능

| 아키텍처 모델 | 주요 구성 요소 | 헤드 수 | 은닉 차원 | 파라미터 규모 | 특징 및 장점 | 예상 Latency (T4 GPU) |
| :---- | :---- | :---- | :---- | :---- | :---- | :---- |
| **TabNet** | Sparse Attention, Steps | \- | 8\~64 | 가변적 | 피처 선택 시각화, 해석 가능성 | \~100ms |
| **TabTransformer** | Categorical Encoder \+ MLP | 4\~8 | 32\~128 | 가변적 | 누락 데이터 강건성, 범주형 처리 | \~50ms |
| **FT-Transformer** | Uniform Tokenizer | 8 | 64\~192 | 가변적 | 피처 상호작용 최적화, SOTA 성능 | 200ms \~ 500ms |
| **SAINT** | Dual Attention (Row/Col) | 8 | 128 | 가변적 | 샘플 간 관계 학습, 높은 일반화 성능 | 500ms+ |
| **TabPFN v2** | In-context Transformer | 12 | 512 | 28M+ | 제로샷 분류, 튜닝 불필요 | 2s \~ 47s (Context) |
| **Sherlock** | Multi-input MLP (1,588 feat) | \- | 가변적 | \- | 시맨틱 타입 탐지 특화 | \~30ms (CPU) |

| 📌 참고. FT-Transformer 동작 원리 (Gorishniy et al., NeurIPS 2021\) • ① Feature Tokenizer: 26개 피처 각각을 독립 임베딩 벡터로 변환 (연속형: 선형, 범주형: 룩업) • ② Self-Attention: 26개 피처 임베딩 \+ \[CLS\] 토큰을 시퀀스로 묶어 Attention → 피처 간 상호작용 학습 • ③ \[CLS\] 출력 벡터: 전체 Tabular의 압축 표현 → FusionLayer에 전달 • ④ 설명 가능성: Attention Map에서 피처 중요도 추출 가능 (어떤 필드가 분류에 기여했는지 시각화) |
| :---- |

## **2.2  모델 병합 (FusionLayer) 전략**

| 📋  \[3/19 회의 결정사항\] • 복잡한 어텐션 매핑 방식은 현재 기술 이해도 및 데이터 품질 수준에서 리스크가 큼. • 각 인코더 출력 벡터를 단순 이어붙이는(Concatenate) Late Fusion을 최우선 베이스라인으로 확정. |
| :---- |

| 구분 | Early Fusion (조기 결합) | 최종 결정 → Late Fusion (지연 결합) |
| :---- | :---- | :---- |
| **결합 시점** | 모델 입력 전 (Feature Level) | 모델 출력 직전 (Decision Level) |
| **데이터 처리** | 텍스트 벡터와 정형 데이터를 하나로 이어 붙여 입력 | 텍스트 전용 모델과 정형 데이터 모델을 각각 운영 |
| **학습 난이도** | 높음 (서로 다른 데이터 타입 간의 스케일 조정 필수) | 낮음 (각 데이터 특성에 맞춰 독립적 최적화 가능) |
| **모델 유연성** | 입력 구조 변경 시 전체 아키텍처 재설계 필요 | RoBERTa 등 특정 모듈의 교체 및 업그레이드가 용이 |
| **적용 전략** | 데이터 간의 복잡한 상관관계를 초기부터 학습 | 텍스트 모델과 정형 모델을 각각 독립적으로 구축한 뒤, 마지막에 두 특징을 합쳐 결론을 내는 방식 (유연한 확장 가능) |
| **장점** | 데이터 간의 잠재적 상호작용 포착에 유리 | 구현이 직관적이며 시스템 유지보수성이 높음 |
| **단점** | 데이터의 형태(차원 수)가 너무 다르면, 특정 데이터에 모델이 편향될 위험 존재 | 데이터 간의 미세한 초기 상호작용 정보 손실 가능성 |

| 🔬  차기 고도화 실험 후보 • Overall Attention: Text 토큰(최대 64\) \+ Tabular 피처(26)를 하나의 시퀀스로 결합 → Cross-Attention. 시퀀스 길이가 짧아 O(N²) 연산 부담 낮음. • MBT (Multimodal Bottleneck Transformer): 소수의 병목 토큰을 매개체로 간접 정보 교환 → 합성 데이터 노이즈 방어. • Dual-stream 앙상블: Text·Tabular 로짓 독립 산출 후 평균. 로짓 불일치 시 Uncertainty 높음으로 간주 → 인간 검토 로직 연동. |
| :---- |

## **2.3  분류 헤드 (ClassificationHead)**

Encoder가 압축한 벡터를 받아 차원을 축소하며 최종 8개 클래스 확률로 변환하는 모듈입니다.

| DeBERTa / KLUE-RoBERTa └── \[CLS\] token representation  (768-dim)    ← 전체 텍스트의 압축 표현      └── Dropout(p=0.1)                       ← 과적합 방지 (합성 패턴 억제)           └── Linear(768 → 256\)                ← 정보 선별 및 차원 압축                └── ReLU                        ← 비선형성 추가                     └── Dropout(p=0.1)          ← 2차 과적합 방지                          └── Linear(256 → 8\)    ← 8개 클래스 점수 산출                               └── Softmax        ← 합이 1인 확률 분포로 변환 |
| :---- |

| 📌 분류 헤드 핵심 설계 원칙 • \[CLS\] 토큰: Self-Attention을 통해 전체 시퀀스 정보가 집약된 벡터 — 분류 태스크의 입력으로 사용 • Dropout(0.1): 학습 시 10% 뉴런 무작위 비활성화 → 합성 데이터 반복 패턴 과적합 방지에 특히 중요 • 768→256→8 구조: 표준적. 단계적 차원 축소로 분류에 불필요한 정보를 걸러내며 압축 • Softmax: 8개 클래스 점수를 합이 1인 확률로 변환 → 예측 신뢰도 수치화 가능 |
| :---- |

## **2.4  학습 최적화 및 파인튜닝 (Training Optimization)**

초기 단계에서 **점진적 동결 해제**(Gradual Unfreezing)를 통해 모델의 기초 체력을 확보하고, 후반부 전 레이어 개방 시 **차등 학습률**(Differential LR)을 적용하여 각 모듈별로 정교한 최적화를 수행하는 전략을 수행.  
**설계 근거:** 학습 초기부터 모든 층에 차등 학습률을 적용할 경우, 미학습 상태의 분류 헤드에서 발생하는 노이즈가 인코더의 정밀한 사전 학습 지식을 오염시켜 전반적인 모델 성능을 저해할 위험이 있음. 따라서 우선적으로 동결 해제를 통해 헤드를 안정화한 뒤, 차등 학습률로 전체적인 가중치 조화를 도모하는 것이 성능 및 학습 안정성 확보 측면에서 가장 유리한 것으로 판단됨.

| 전략 | 설명 | 설정값 |
| :---- | :---- | :---- |
| **점진적 동결 해제 (Gradual Unfreezing)** | 초기 Head만 학습 → 이후 상위 Encoder 레이어부터 단계적 동결 해제 (망각 현상 방지를 위함) | \- |
| **차등 학습률 (Differential LR)** | 인코더 레이어와 분류 헤드에 다른 학습률 적용 \- 사전 학습된 지식을 보존하면서, 새로운 작업에 빠르게 적응하기 위함 \- Encoder (낮은 LR): 기존 지식을 잊어버리지 않기 위함 (조절시, 2, 5배 단위 사용) \- Head (높은 LR): 정오탐 분류를 빠르게 배우기 위함, Encoder보다 약 50\~100배 크게 설정 | \- Encoder: 보통 1e-5 \~ 5e-5 \- Head: 1e-4 \~ 1e-3 |
| **손실 함수 (Loss Function)** | 초기: CrossEntropyLoss → 조건부: Focal Loss 전환(Per-class F1 현저히 낮은 클래스 발생 시 전환)  | Per-class F1 \< 0.60 |
| **옵티마이저 (Optimizer)** | AdamW (Weight Decay 포함 Adam 변형) \- 먼저 LR을 조절하여 모델이 수렴하는지 확인 \- 수렴 후, 검증 데이터 점수가 떨어진다면 weight\_decay 높임 \- **Batch Size**를 조절하여 Gradient 계산의 안정성을 확보 | 초기 설정: LR=2e-5, weight\_decay=0.01 |

**📌 참고. CrossEntropyLoss와 Focal  Loss 비교**

| 구분 | CrossEntropyLoss (기본) | Focal Loss (심화) |
| :---- | :---- | :---- |
| **핵심 철학** | "모든 정답을 잘 맞히자" | "어렵고 헷갈리는 것들을 잡아내자" |
| **데이터 비중** | 데이터가 많은 클래스에 휘둘리기 쉬움 | 데이터가 적거나 어려운 클래스에 집중함 |
| **사용 시점** | 학습 초기 (Phase 1 시작) | 특정 클래스 F1 점수가 낮을 때 (고도화) |
| **효과** | 전체적인 정확도 향상 | 오탐(FP)과 정탐(TP)의 미세 분류 성능 향상 |

## **2.5  평가 지표 및 실험 추적 (Metrics & Experiment Tracking)**

**평가 지표 (MetricsTracker)**

| 지표 | 계산 방법 | 목적 및 해석 |
| :---- | :---- | :---- |
| **Macro F1** | 각 클래스 F1의 단순 평균 \= (F1\_0+...+F1\_7)/8 | 전체 성능 지표. FP-7처럼 어려운 클래스 성능 저하 시 전체 수치에 반영됨. |
| **Per-class F1** | F1 \= 2×(Precision×Recall)/(P+R) 클래스별 산출 | 어느 클래스가 취약한지 진단. 표면적으로 유사한 클래스 간 혼동 파악. |
| **Confusion Matrix** | 8×8 행렬 — 행: 실제, 열: 예측 | 특정 FP 클래스 간 오분류 패턴 시각화. 대각선 외 큰 값 \= 모델이 혼동하는 클래스 쌍. |

**실험 추적 예시 (Experiment Tracking)**

| experiments/ ├── config/ │   ├── baseline\_tfidf.yaml          \# TF-IDF \+ XGBoost 베이스라인 │   ├── exp\_001\_deberta.yaml          \# DeBERTa 실험 설정 │   └── exp\_002\_kcelectra.yaml        \# KcELECTRA 실험 설정 └── runs/     └── exp\_001/         ├── model.pt                  \# 저장된 모델 가중치         ├── metrics.json              \# Macro F1 \+ Per-class F1 기록         ├── confusion\_matrix.png      \# 오분류 패턴 시각화         └── config\_snapshot.yaml      \# 해당 실험 하이퍼파라미터 스냅샷 |
| :---- |

| 📌 실험 추적 핵심 원칙 • 합성 데이터 버전 \+ 전처리 규칙 버전 \+ 모델 설정을 항상 함께 기록 • YAML 파일로 모든 하이퍼파라미터 고정 → 동일 실험 재현 보장 • MLflow 또는 Weights & Biases 연동 권장 — 여러 실험 시각적 비교 가능 • Confusion Matrix 매 실험마다 자동 생성 → 클래스별 오분류 패턴 추이 관리 |
| :---- |
