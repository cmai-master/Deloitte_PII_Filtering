# **0\. 문서 개요**

본 문서는 LG U+ Server-i 플랫폼의 PII 오탐·정탐 분류 모델 구축을 위한 평가기준 파이프라인의 상세 설계서입니다. 합성 데이터 파이프라인(v0.4), 데이터 전처리 파이프라인(v1.0), DL 모델 파이프라인(v1.0)과 연계하여, 각 단계의 산출물에 대한 통합 평가 체계 및 배포 가능 여부 판별 기준을 정의합니다.

| 📌 설계 범위 및 전제 • 평가 대상: 합성 데이터 품질, 전처리 결과, 학습된 DL 모델의 추론 성능 • 분류 목표: TP(정탐) 1종 \+ FP(오탐) 7종, 총 8개 클래스 • 핵심 기준: TSTR(Train on Synthetic, Test on Real) F1 ≥ 0.75 (Hard Gate) • 연계 문서: 합성데이터 생성 파이프라인 v0.4 / 데이터 전처리 파이프라인 v1.0 / DL모델 파이프라인 v1.0 |
| :---- |

# **1\. 평가 체계의 계층 구조 (Metrics Hierarchy)**

단순 지표 관리를 넘어, 실제 배포 가능 여부를 결정하는 'Gatekeeper' 지표를 강화합니다. 각 계층의 지표는 파이프라인 전 과정(합성 데이터 → 전처리 → 모델 학습 → 배포)에 걸쳐 단계별로 적용됩니다.

| 분류 | 지표 | 목적 및 비즈니스 가치 | 비고 |
| :---- | :---- | :---- | :---- |
| Primary | TSTR F1-Score | 실데이터(Real-Data Holdout)에서의 성능. 0.75 미만 시 배포 차단. | Hard Gate |
| Diagnostic | Per-class F1 | 특정 FP 클래스(예: FP-7 맥락 오탐)의 성능 저하 진단 및 Focal Loss 전환 결정. | — |
| Consistency | Tri-Consistency | 클래스-Text-Tabular 간 논리적 일관성 오류율 측정. 합성 데이터 품질 피드백 루프에 활용. | Feedback |
| Data Quality | Binary Checklist 통과율 | 합성 데이터 생성 단계의 7개 항목 바이너리 평가 통과 비율. 전체 통과 시만 학습 데이터 편입. | Feedback |

| 📌 Hard Gate 기준 설정 근거 • TSTR F1 0.75: 합성 데이터 파이프라인(v0.4) 5.3절 핵심 KPI 기준과 통일. 합성 F1과 Real F1 격차 15% 이상 시 합성 데이터 재생성 프로세스 강제 회귀. |
| :---- |

# **2\. 평가 파이프라인 워크플로우 (Automated Workflow)**

평가는 합성 데이터 품질 검증(Step 0)부터 모델 성능 평가(Step 1\~2)까지 순차적으로 수행됩니다. 각 단계는 이전 단계의 Gate를 통과해야 다음 단계로 진행됩니다.

## **Step 0: 합성 데이터 품질 평가 (Binary Checklist)**

모델 학습 이전에, 생성된 합성 데이터 각 레코드를 별도 LLM(Judge)이 바이너리(O/X) 방식으로 평가합니다. 생성 LLM과 Judge LLM은 반드시 다른 모델을 사용하여 자기평가 편향을 제거합니다.

| 📋 \[연계\] 합성 데이터 파이프라인 v0.4 — 5.1절 LLM-as-a-Judge 검증 • 생성용: Claude Haiku (합성 데이터 생성) / 평가용: GPT-5.4 mini (Judge) — 교차 검증 구조 • 전체 7개 항목 모두 O 판정 시에만 학습 데이터셋 편입. 탈락 레코드는 Rejected\_Pool로 분리 관리. |
| :---- |

| \# | 평가 항목 | O 판단 기준 요약 |
| :---- | :---- | :---- |
| (1) | PII 마스킹 패턴 실데이터 Rule 준수 여부 | 한국 번호 형식·도메인 규칙에 맞는 마스킹 패턴 생성 |
| (2) | Tabular 필드값의 현실적 시나리오 부합 여부 | 26개 필드가 클래스 맥락과 논리적으로 일치 |
| (3) | 클래스-Text-Tabular 삼각 일관성 (현재: 보류) | 세 컴포넌트 간 모순 없이 일치. 실 Tabular 데이터 확보 후 적용 |
| (4) | 스키마 필드 누락 및 타입 오류 여부 | JSON 스키마 필드 완비, 데이터 타입 정확 |
| (5) | 완전 일치 기준 중복 레코드 여부 | 기존 DB와 모든 필드값이 완전히 동일하지 않음 |
| (6) | 프롬프트 세부 지침 엄격 준수 여부 | JSON 외 불필요한 텍스트 없음, 제약 조건 모두 충족 |
| (7) | 배치 내 패턴 다양성 및 의미적 중복 방지 | 핵심 패턴 과도 반복 없음, Mode Collapse 발생 아님 |

| 📌 Rejected Pool 처리 원칙 • 탈락 레코드는 즉시 삭제하지 않고 Rejected\_Pool(Edge\_Case\_DB)로 이관. • Discard\_Reason \+ Prompt\_ID 태깅 후, Stress Test 및 프롬프트 엔지니어링 피드백 루프에 활용. • Hard Negative 테스트셋 구축: 탈락 레코드로 모델이 비정상 입력에 오작동하지 않는지 검증. |
| :---- |

## **Step 1: Multi-Source Evaluation (TSTR 중심)**

학습이 완료된 모델을 두 가지 데이터 소스로 평가하여, 합성 데이터 기반 모델의 실데이터 적용 가능성을 검증합니다.

| 평가 소스 | 목적 및 운영 기준 |
| :---- | :---- |
| Synthetic Val (20%) | 학습 중 Overfitting 방지 및 하이퍼파라미터 최적화 모니터링. |
| Real-Data Holdout (Test) | 실제 운영 환경 모사 능력 최종 검증. TSTR F1 0.75 미만 시 배포 차단(Hard Gate). |

| 📌 Synthetic F1 vs Real F1 격차 모니터링 • Synthetic F1과 Real F1의 격차가 15% 이상 발생할 경우, 모델 재학습이 아닌 합성 데이터 생성 프롬프트(Scenario) 재검토 프로세스로 강제 회귀. • 이 기준은 합성 데이터 자체의 품질 문제(분포 편향, 시나리오 다양성 부족)를 조기에 포착하기 위한 핵심 Trigger. • 분석 기준: 전처리 파이프라인(v1.0) 1.6절 데이터 분할 기준(Train 80% / Val 20% / Test-Real)에 따라 산출. |
| :---- |

## **Step 2: Threshold-based Decision & Feedback**

학습 결과에 따라 자동으로 후속 액션을 결정하는 분기 로직입니다.

| 트리거 조건 | 조치 유형 | 상세 액션 |
| :---- | :---- | :---- |
| Per-class F1 \< 0.60 (특정 클래스) | Focal Loss 전환 | CrossEntropyLoss → Focal Loss 자동 적용. 어렵고 희귀한 클래스(FP-7 등) 집중 학습. |
| 특정 Prompt\_ID 오답률 이상 급등 | Prompt 품질 경고 | 해당 Prompt\_ID를 'Low Quality'로 마킹. 합성 데이터 생성 파이프라인에 경고 발송 및 재설계 요청. |
| Synthetic \- Real F1 격차 ≥ 15% | 데이터 재생성 강제 회귀 | 모델이 아닌 합성 데이터 Scenario 프롬프트 재검토. 합성 데이터 파이프라인 v0.4 4.2절 시나리오 재설계. |
| TSTR F1 \< 0.75 | 배포 차단 (Hard Gate) | 배포 불가 판정. 합성 데이터 볼륨·품질 또는 모델 아키텍처 전면 재검토. |

# **3\. 데이터 계보 및 실험 통합 관리 (Lineage Tracking)**

합성 데이터의 생성 원천과 모델 성능을 하나로 묶어 관리하여 '최적의 데이터 레시피'를 도출합니다. 합성 데이터 파이프라인(v0.4) 2.5절의 계보 메타데이터(Prompt\_ID, Model\_Version 등)를 평가 결과와 연계 추적합니다.

| Tracking 항목 | 설명 및 연계 정보 |
| :---- | :---- |
| Experiment\_ID | 실험 고유 식별자. DL모델 파이프라인 2.5절 runs/ 디렉토리와 1:1 매핑. |
| Model\_Ver | 사용된 TextEncoder 모델 버전 (예: mDeBERTa-v3-base-v1.0). |
| Data\_Ver | 학습에 사용된 전처리 데이터 버전 (예: processed/v1.0/). 전처리 파이프라인 1.6절 버전 관리 기준. |
| Prompt\_ID | 합성 데이터 생성에 사용된 프롬프트 ID (예: Prompt\_v1.2\_TP1). 합성 데이터 파이프라인 2.5절 계보 메타데이터. |
| TSTR\_F1 | Real-Data Holdout 기반 최종 F1 점수. Hard Gate(0.75) 통과 여부 기록. |
| Synthetic\_Real\_Gap | Synthetic F1 \- Real F1 격차. 15% 이상 시 데이터 재생성 Trigger. |

| 📌 Prompt-wise Analysis 예시 • \[예시\] 프롬프트 시나리오 A(내부망 중심)로 생성된 데이터가 FP-4(도메인 오탐) 클래스 성능에 결정적인가? — 데이터 계보를 통해 자동 분석. • Prompt\_ID별 Per-class F1 분포를 교차 분석 → 특정 Scenario가 특정 클래스 성능을 좌우하는 패턴 식별. • 실험 추적 도구: MLflow 또는 Weights & Biases 연동 권장 (DL모델 파이프라인 2.5절 동일 기준). |
| :---- |

# **4\. 심화 오류 분석 (Deep Error Analysis)**

모델의 오답이 '학습 부족'인지 '데이터 모순'인지 판별합니다. 원인 진단 없이 단순히 학습을 반복하는 낭비를 방지합니다.

## **4.1 삼각 일관성 검증 (Tri-Consistency Verification)**

Confusion Matrix에서 오답으로 분류된 샘플의 Text · Tabular · Class 간 논리적 일관성을 LLM을 통해 재검증합니다.

| 검증 단계 | 상세 내용 |
| :---- | :---- |
| ① 오답 샘플 추출 | Confusion Matrix에서 대각선 외 값이 큰 셀(혼동 클래스 쌍)의 샘플 선별. |
| ② LLM 재검증 | 추출된 샘플의 Text 패턴, Tabular 피처가 레이블 클래스 정의와 일치하는지 Judge LLM으로 검증. |
| ③ 원인 분류 | 모델 학습 부족(→ 학습 강화) vs 데이터 자체 모순(→ 생성 파이프라인 정제 규칙 업데이트) 구분. |
| ④ 제외 처리 | 마스킹 오류 등 데이터 자체 모순 케이스는 모델 성능 평가에서 제외 후 Rejected\_Pool 이관. |

| 📌 Tri-Consistency 현재 상태 • 클래스-Tabular 일관성 검증은 실 Tabular 데이터 확보 전까지 자동화 보류 (합성 데이터 파이프라인 v0.4 2.1절 동일 조건). • 현재 단계: Text ↔ Class 일관성 중심으로 운영. Tabular 확보 후 전체 삼각 검증으로 확장. |
| :---- |

## **4.2 클래스별 오분류 패턴 분석**

표면적으로 유사한 FP 클래스 간 혼동 패턴을 체계적으로 추적합니다.

| 혼동 클래스 쌍 | 주요 혼동 원인 | 대응 방안 |
| :---- | :---- | :---- |
| FP-2 ↔ FP-4 | 이메일 패턴 동일 (@도메인). 도메인이 내부 여부로 구분. | MaskTokenEncoder 도메인 유형 피처 강화. Tabular IP 대역 피처 활용. |
| FP-3 ↔ FP-5 | 숫자 자리수 유사 (날짜 vs 바이트). 마스킹 길이로 구분. | MaskTokenEncoder \[MASK\_N\] 길이 정보 전처리 정밀도 확인. 컨텍스트 강화. |
| FP-7 ↔ TP | 컨텍스트 의존적 분류. 패턴만으로 구분 불가. | 앞뒤 10자 컨텍스트 품질 검증. 합성 시나리오 다양성 확대. |
| FP-1 ↔ FP-3 | 숫자 나열 패턴 유사 (코드 vs 타임스탬프). | 컨텍스트 키워드(xpiryDate, VERSION 등) 생성 시나리오 강화. |

# **5\. 평가 파이프라인 디렉토리 구조 (Evaluation Artifacts)**

모든 평가 산출물을 실험 단위로 관리하여 재현성을 보장합니다. DL모델 파이프라인(v1.0) 2.5절 실험 추적 구조와 통합됩니다.

| 📌 실험 추적 핵심 원칙 (DL모델 파이프라인 2.5절 확장) • 합성 데이터 버전(Data\_Ver) \+ 전처리 규칙 버전 \+ 모델 설정 \+ Prompt\_ID를 항상 함께 기록. • Hard Gate 통과/차단 결과를 gate\_log에 자동 기록 → 배포 감사 추적(Audit Trail) 역할. • Confusion Matrix 매 실험마다 자동 생성 → 클래스별 오분류 패턴 추이 관리. • MLflow 또는 Weights & Biases 연동으로 여러 실험 시각적 비교. |

# **6\. 미결 항목 및 의존성 관리 (Open Issues)**

평가 파이프라인의 일부 항목은 전처리 파이프라인 및 실 데이터 확보 여부에 따라 결정이 보류되어 있습니다.

| \# | 미결 항목 | 의존 조건 | 우선순위 |
| :---- | :---- | :---- | :---- |
| 1 | Tri-Consistency 자동화 (클래스-Tabular 검증) | 실 Tabular 데이터 확보 후 클래스-Tabular 상관관계 검증 가능 | High |
| 2 | Max Length 최적 기준 확정 (32 / 64 / 128\) | 합성 데이터 전량 생성 후 토큰 길이 분포 EDA 수행 필요 (전처리 파이프라인 1.3절) | High |
| 3 | Truncation 방향 결정 (pre / post) | 실데이터에서 핵심 분류 단서 위치 분석 필요 (전처리 파이프라인 1.2절 미결사항) | Medium |
| 4 | Real-Data Holdout 최소 확보 수량 기준 정의 | TSTR 평가의 신뢰도를 보장할 실데이터 최소 건수 미정 — 클래스별 균형 고려 필요 | High |

# **7\. \[부록\] 추후 고도화 아이디어 (Stress Test)**

향후 시스템의 안정성과 견고성을 극대화하기 위해 다음의 고도화 테스트 단계를 추가할 수 있습니다.
