# **대규모 언어 모델(LLM) 기반 합성 데이터 생성 파이프라인: 심층 분석 및 아키텍처 설계 보고서**

## **1\. LLM 기반 데이터 생성의 핵심 리스크**

생성형 모델의 확률적 특성과 재귀적 사용은 기존 파이프라인과 다른 치명적 시스템 리스크를 유발함.

### **1.1 모델 붕괴 (Model Collapse)**

* **정의:** AI 생성 데이터로 후속 AI를 반복 학습시킬 때, 모델 성능이 점진적·비가역적으로 저하되는 현상.  
* **원인:** 모델이 희귀 데이터(꼬리 분포)를 배제하고 최빈값(Mode) 위주로 생성하여 다양성이 소실됨.  
* **진행 과정:** 소수 집단 및 희귀 표현을 상실하는 초기 붕괴를 거쳐, 현실과 동떨어진 단순화된 패턴만 남는 후기 붕괴로 악화됨.  
* **대응:** 인간 생성 실제 데이터(Real Data)의 비율 유지 및 외부 엔트로피 주입 아키텍처 필요.

### **1.2 내부 호출 편향 (Intra-Call Bias) 및 분포 수축**

* **정의:** 비용 최적화를 위해 단일 API 호출로 다수의 샘플을 생성할 때, 샘플들이 서로 독립성을 잃고 앞선 생성물의 패턴에 종속되어 획일화되는 현상.  
* **원인:** 모델이 이전 출력물을 필수 문맥(Context)으로 참조하려는 자기회귀적(Autoregressive) 특성과, 어텐션 가중치가 특정 토큰에 집중되는 비혼합적(Non-mixing) 동역학 때문임.  
* **리스크:** 샘플 간 스타일이 동질화되고 순서에 따라 지시 준수율이 감쇠(Lost in the Middle)하며, 결과적으로 꼬리 분포(Tail Distribution) 데이터가 소실되어 모델 붕괴가 가속화됨.

### **1.3 합성 아티팩트 (Synthetic Artifacts) 및 과적합 리스크**

* **정의:** 과도하게 정제된 데이터는 모델의 분포 붕괴(Model Collapse)를 유발하고 실제 사용자 환경의 불규칙한 입력에 대한 대응 능력을 저하시키는 '합성 아티팩트(Synthetic Artifacts)' 문제를 야기함.   
* **원인:** 합성 데이터에만 의존할 경우, 모델은 생성 모델 특유의 '매끄럽고 평균적인' 텍스트 분포에 과적합되는 경향이 있음.   
* **리스크:** 모델이 생성하는 콘텐츠의 다양성이 줄어들고, 실제 인간 언어의 특징인 불규칙성, 비문, 독창적인 문체에 대한 이해도가 떨어져 모델 붕괴로 이어짐.

### **1.4 다양성 부족 (Mode Collapse)**

* **프롬프트 편향:** LLM이 주어진 소수 예시(Few-shot)의 길이와 구조를 지나치게 모방하여 동질화된 데이터를 양산함.  
* **리스크:** 의미론적 다양성이 결여된 데이터로 학습 시, 아웃라이어(OOD) 입력에 대한 모델 성능이 급격히 저하됨.

## **2\. 문제 해결을 위한 방법론적 솔루션**

앞서 제기된 난제들을 극복하기 위해, 최신 연구들은 다음과 같은 고도화된 방법론들을 제안하고 있다.

### **2.1 혼합 비율의 최적화: 30/70 법칙**

모델 붕괴를 막기 위한 가장 확실한 방법은 합성 데이터만으로 학습하지 않는 것이다. 2025년 발표된 연구들에 따르면, 약 30\~33%의 고품질 합성 데이터와 67\~70%의 리얼 데이터(Organic Data)를 혼합했을 때 가장 우수한 성능을 보인다.

* **역할 분담:** 합성 데이터는 명확한 논리 구조, 추론 과정, 깨끗한 포맷을 제공하는 "교과서" 역할을 하며, 리얼 데이터는 언어적 다양성과 현실의 노이즈를 학습시키는 "현장 실습" 역할을 수행한다.  
* **웹 데이터의 재구성(Rephrasing):** 리얼 데이터(웹 문서 등)를 단순히 사용하는 대신, LLM을 이용해 내용을 유지하되 형식을 정제(Refining)하여 사용하는 방식이 순수 합성 데이터보다 모델 붕괴 저항성이 높다.

#### **2.1.1 \[제로 리얼 데이터(Zero Real Data) 환경에서의 100% 합성 데이터 대안 옵션\]** 실제 데이터를 70% 이상 확보할 수 없는 제로 리얼 데이터 환경에서는 다음의 방법론들을 파이프라인에 단계적으로 구현하여 30:70 법칙의 한계를 우회해야 합니다.

**대안 아키텍처 1: 교과서적 품질(Textbook Quality) 큐레이션 및 지시 역전 파이프라인** 실제 데이터의 부재를 '이상적인 고품질 데이터' 설계로 대체하는 접근 방식입니다.

* **Step 1\. 시드 큐레이션 (랜덤 포레스트 기반):** GPT-4를 활용해 10만 개 규모의 공개 데이터(Wiki, arXiv 등)에 '교육적 가치' 점수를 부여하여 랜덤 포레스트 분류기를 학습시킵니다. 이후 이 분류기로 수십억 개의 원시 데이터를 필터링하여 최상위 품질의 시드만 추출합니다.  
* **Step 2\. 개념 추출 및 교과서 생성:** 추출된 시드 데이터에서 핵심 개념을 분리한 후, 교사 모델(Teacher Model)을 통해 다양한 난이도의 '합성 교과서 챕터'를 생성합니다.  
* **Step 3\. 지시 역전 (Instruction Reversal):** 생성된 텍스트나 코드를 '답변'으로 먼저 고정시키고, "이 답변을 유도할 수 있는 프롬프트를 역으로 생성하라"고 지시하여 프롬프트의 다양성을 인위적으로 극대화합니다.  
* **Step 4\. 자기 수정 워크플로우:** 파이프라인 최종단에 비평 프롬프트를 배치하여 생성된 데이터의 논리적 오류나 사실관계 결함을 비평하고 스스로 수정(Self-Revision)하도록 강제합니다.

**대안 아키텍처 2: 진화적 복잡성 엔지니어링 (Evol-Instruct) 파이프라인** 모델이 평이한 답변으로 수렴하는 '평균 회귀' 현상을 막기 위해 데이터의 난이도를 강제로 상향시킵니다.

* **Step 1\. 초기 시드 주입:** 단순하고 명확한 기초 지시사항을 생성하여 Epoch 1 학습에 할당합니다.  
* **Step 2\. 심층 진화 (In-Depth Evolution):** 기존 프롬프트에 "외부 라이브러리 없이 시간 복잡도 O(n)을 만족할 것"과 같은 제약 조건을 추가하거나 , 현상의 근본 원인 설명을 요구하는 등 추론 단계를 명시적으로 증가시킵니다.  
* **Step 3\. 광폭 진화 (In-Breadth Evolution):** 기존 문맥을 완전히 새로운 도메인(예: "재고 관리 시스템" → "유전자 서열 분석 시스템")으로 변이(Mutation)시켜 데이터셋의 커버리지를 넓힙니다.  
* **Step 4\. 커리큘럼 학습 적용:** 진화 단계에 따라 Epoch 2(심화), Epoch 3(고난도)로 데이터를 분류하여 순차적으로 학습시킵니다.

**대안 아키텍처 3: 에이전트 기반 시뮬레이션 (Matrix Framework)** 단일 모델의 독백(Monologue)이 가진 한계를 벗어나, 인간의 상호작용성을 멀티 에이전트 시뮬레이션으로 구현합니다.

* **Step 1\. 사용자 에이전트 생성:** 사용자 에이전트에게 "성격 급한 고객", "기초가 부족한 학생" 등 무작위 페르소나를 부여하여 엣지 케이스 질문을 유도합니다.  
* **Step 2\. 전문가 에이전트 생성:** 타겟 모델이나 더 강력 교사 모델이 해당 질문에 대한 1차 답변을 생성합니다.  
* **Step 3\. 비평가 에이전트(Critic) 개입:** 제3의 비평가 에이전트가 답변의 모호성이나 제약 조건 위반 여부를 검증하고 피드백을 전달하여 RLAIF 루프를 형성합니다.  
* **Step 4\. 협력적 추론 기록:** 수학이나 코딩 등 복잡한 태스크의 경우, 에이전트 간의 가설 수립 및 오류 수정 등 토론 과정 자체를 저장하여 모델에 '문제 해결 과정(Process Supervision)'을 학습시킵니다.

**대안 아키텍처 4: 노이즈 정규화 및 피벗 토큰 최적화 (NEFTune & PTS)** 합성 데이터 특유의 지나치게 매끄러운 텍스트(Over-smoothing)에 모델이 과적합되는 것을 방지합니다.

* **Step 1\. NEFTune 노이즈 주입:** 순전파(Forward Pass) 과정에서 입력 토큰 임베딩 X\_emb에 균일 분포에서 추출한 무작위 노이즈 epsilon을 더합니다.![][image1] 노이즈 크기 ![][image2] 는 약 ![][image3] 로 설정합니다.  
* **Step 2\. 피벗 토큰(Pivotal Token) 탐색:** 모델의 논리적 전개가 성패를 가르는 확률적 분기점(피벗 토큰)을 식별합니다.  
* **Step 3\. DPO 쌍 구축:** 해당 피벗 토큰을 기준으로 긍정적 예시와 부정적 예시를 생성한 뒤, 직접 선호 최적화(DPO)를 통해 의사결정 경로를 집중 학습시킵니다.

**대안 아키텍처 5: 역헌법적 AI (Inverse Constitutional AI) 정렬** 인간의 피드백(RLHF)이 불가능한 환경에서 모델이 스스로의 품질 기준을 수립하게 만듭니다.

* **Step 1\. 헌법(Principles) 추출:** 소량의 고품질 시드나 가이드라인을 역공학하여 "간결성", "보수성" 등의 도메인 핵심 원칙을 추출합니다.  
* **Step 2\. 자율 비평 (Self-Critique):** 생성된 전체 합성 데이터를 1단계에서 추출한 헌법에 대조하여 모델이 스스로 결함을 비평하게 합니다.  
* **Step 3\. 선호 데이터셋 구축:** 비평을 반영해 수정된 데이터(Chosen)와 원본 데이터(Rejected)를 매핑하여 자체 선호 데이터셋을 구축합니다.  
* **Step 4\. PPO/DPO 파인튜닝:** 구축된 데이터셋을 통해 정렬 학습을 수행하여, 인간 개입 없이도 시간이 지날수록 품질이 향상되는 자기 수정 루프를 완성합니다.

### **2.2 내부 호출 편향(Intra-Call Bias) 완화 전략**

단일 API 호출 내에서 발생하는 어텐션 간섭과 다양성 수축을 방지하기 위해 6가지 아키텍처 및 샘플링 최적화 전략을 단계적으로 파이프라인에 적용합니다.

**대안 아키텍처 1: 독립적 병렬 디코딩 (Skeleton-of-Thought)** 단일 프롬프트로 다수의 샘플을 직렬로 길게 생성할 때 발생하는 문맥 오염을 차단하고 지연 시간(Latency)을 단축합니다.

* **Step 1\. 뼈대(Skeleton) 추출:** LLM에게 최종 답변 대신 "생성할 데이터 10개의 핵심 주제(Bullet point)"만 먼저 짧게 출력하도록 지시합니다.  
* **Step 2\. 병렬 API 호출:** 추출된 10개의 주제를 각각 독립된 API 호출(또는 워커 노드)로 분리하여 동시에 전송합니다.  
* **Step 3\. 결과 병합:** 각 워커에서 이전 생성물의 어텐션 간섭 없이 독립적으로 생성된 결과를 최종 데이터셋으로 병합합니다.

**대안 아키텍처 2: 상관 표본 추출 (Correlated Sampling, CorrSynth)** 확률적 디코딩 과정에 개입하여 기존에 생성된 데이터의 분포와 의도적으로 반대되는(Anti-correlated) 토큰을 선택하게 강제합니다.

* **Step 1\. 로짓(Logit) 스냅샷 저장:** 첫 번째 샘플을 생성할 때 선택된 토큰들의 확률 분포(Logits)를 메모리에 캐싱합니다.  
* **Step 2\. 디코딩 타임 패널티 부여:** 두 번째 샘플 생성 시, 다음 토큰을 예측할 때 캐싱된 로짓과 비교하여 앞서 빈번하게 선택된 토큰의 가중치에 동적 페널티를 부여합니다.  
* **Step 3\. 다양성 극대화:** 이를 통해 동일한 프롬프트라도 확률적 분기점에서 전혀 다른 어휘와 구조를 선택하게 만들어 데이터셋의 엔트로피를 방어합니다.

**대안 아키텍처 3: 어텐션 마스킹 분리 (Attention Masking Isolation)** 오픈소스 모델을 자체 서빙할 때, KV 캐시(Key-Value Cache) 공유로 인한 토큰 간섭을 하드웨어 레벨에서 차단합니다.

* **Step 1\. 배치(Batch) 구성:** 여러 개의 데이터 생성 요청을 묶어 하나의 배치 연산으로 GPU 메모리에 올립니다.  
* **Step 2\. 커스텀 어텐션 마스크 적용:** 2D 어텐션 마스크 행렬을 조작하여, 배치 내의 N번째 시퀀스가 N-1번째 시퀀스의 토큰을 참조(Attend)하지 못하도록 0으로 차단합니다.  
* **Step 3\. 독립적 VRAM 연산:** 연산 효율성을 유지하면서도 각 시퀀스가 수학적으로 완벽히 격리된 상태로 독립적 생성을 수행하도록 보장합니다.

**대안 아키텍처 4: 단일 호출 내 청크 분할 (Chunked Batch Generation)** 비용과 속도를 위해 한 번에 너무 많은 데이터를 요구할 경우 발생하는 어텐션 소실(Lost in the middle)을 방지합니다.

* **Step 1\. 임계점(Threshold) 프로파일링:** 타겟 모델이 앞선 문맥에 동화되어 획일화된 출력을 내뱉기 시작하는 최대 생성 임계 개수(예: 5개)를 실험적으로 찾아냅니다.  
* **Step 2\. 루프 오케스트레이션:** "100개 생성" 요청이 들어오면 파이프라인 오케스트레이터가 이를 "5개 생성 x 20회 반복" 워크플로우로 자동 분할(Chunking)합니다.  
* **Step 3\. 비동기 큐 처리:** 분할된 워크로드를 비동기 메시지 큐에 담아 여러 모델 인스턴스에 분산 처리시켜 편향 누적을 끊어냅니다.

**대안 아키텍처 5: 동적 하이퍼파라미터 셔플링 (Dynamic Hyperparameter Shuffling)** 고정된 디코딩 파라미터가 초래하는 데이터 구조의 획일화를 시스템 차원에서 회피합니다.

* **Step 1\. 파라미터 풀(Pool) 정의:** 생성을 요청할 때마다 고정값을 쓰지 않고, Temperature, Top-p, Top-k의 허용 범위를 넓게 정의합니다.  
* **Step 2\. 난수 기반 주입:** 파이프라인의 API 호출부에서 매 호출마다 난수 발생기를 통해 추출한 하이퍼파라미터 조합을 헤더에 무작위로 주입합니다.  
* **Step 3\. 다형성 확보:** 샘플링의 무작위성 기준 자체가 계속 변동하므로, 보수적인 답변과 창의적인 답변이 자연스럽게 혼합된 데이터셋을 획득합니다.

**대안 아키텍처 6: 문자열 사고 종자 기반 프롬프트 앵커링 (String Seed of Thought)** 모델의 출력 관성을 물리적으로 비틀어 연속적 패턴 고착화를 파괴합니다.

* **Step 1\. 난수 시드 생성:** 파이프라인에서 매 샘플링 직전 8자리의 무작위 영문/숫자 조합(예: "X7K2P9M")을 생성합니다.  
* **Step 2\. 프롬프트 헤더 강제:** "응답의 첫 단어로 반드시 \[X7K2P9M\]를 출력한 후 내용을 시작할 것"이라는 시스템 프롬프트를 주입합니다.  
* **Step 3\. 문맥 초기화 및 후처리:** 모델이 무작위 문자열을 출력하는 순간 이전 문맥의 관성이 일시적으로 리셋되며 전혀 다른 토큰 스트림이 유도됩니다. 이후 데이터 정제 단계에서 앵커 문자열을 정규식으로 일괄 삭제합니다.

### **2.3 자연스러운 노이즈 주입 및 일반화 향상 전략**

모델의 과적합을 방지하고 일반화 능력을 극대화하기 위해 다음 5가지 노이즈 주입 전략을 파이프라인에 적용합니다. 

**대안 아키텍처 1: 임베딩 레벨 정규화 (NEFTune)** 텍스트 자체를 수정하지 않고 모델의 내부 표현에 직접적으로 노이즈를 주입합니다. 

* **Step 1\. 순전파 단계 개입:** 파인튜닝 과정의 순전파 단계에서 입력 토큰 시퀀스가 주어졌을 때, LLM은 이를 임베딩 벡터로 변환합니다.   
* **Step 2\. 무작위 노이즈 추가:** 변환된 임베딩 벡터에 균일 분포에서 추출한 무작위 노이즈를 추가합니다.   
* **Step 3\. 과적합 방지 유도:** 이 과정은 모델이 학습 데이터의 정확한 임베딩 위치에 과적합되는 것을 방지하고, 임베딩 공간 내의 작은 변동에도 견고한 특징을 학습하도록 강제합니다. 

**대안 아키텍처 2: 유형 인식 데이터 증강 (TypeDA)** LLM을 정교한 주석가로 활용하여 특정 유형의 문법적 오류를 의도적으로 주입합니다. 

* **Step 1\. 오류 유형 선정 및 마스킹:** 현실적인 분포에 기반하여 세분화된 문법 오류 유형을 선택하고, 해당 오류를 적용할 위치를 마스킹합니다.   
* **Step 2\. 유형 인식 프롬프팅:** LLM에게 문맥 학습을 활용하여 원본 문장의 의미를 보존하면서도 자연스러운 오류가 포함되도록 마스킹된 부분을 채우라고 지시합니다.   
* **Step 3\. 품질 검증 및 필터링:** ERRANT 툴킷을 사용하여 생성된 오류가 의도한 문법적 범주에 속하는지 검증하고, 레벤슈타인 거리를 측정하여 구조적 유사성 훼손 여부를 확인합니다. 

**대안 아키텍처 3: 적대적 데이터 생성 (AdvERSEM)** 모델의 논리적 허점이나 사실관계 파악 능력을 공격하는 '적대적 데이터'를 생성합니다. 

* **Step 1\. 그래프 변환 및 조작:** 텍스트를 추상 의미 표현(AMR) 그래프로 변환한 후, 주체 변경이나 불확실성 주입 등의 의미적 조작을 가합니다.   
* **Step 2\. 적대적 텍스트 재생성:** 변형된 그래프를 다시 자연어로 변환하여, 교묘한 조작이나 편향된 주장이 포함된 데이터를 생성합니다.   
* **Step 3\. 탐지 능력 강화 학습:** 생성된 적대적 데이터로 모델을 학습시켜 미묘한 의미 변화나 왜곡을 탐지하는 능력을 향상시킵니다. 

**대안 아키텍처 4: 에이전트 기반 다양성 스캐폴딩 (MetaSynth)** 여러 '전문가 에이전트'가 협업하여 데이터를 생성하는 메타 프롬프팅 프레임워크를 적용합니다. 

* **Step 1\. 에이전트 오케스트레이션:** 중앙 오케스트레이터 모델이 데이터 생성 작업을 여러 에이전트에게 분배합니다.   
* **Step 2\. 페르소나 할당:** 각 에이전트에게 '회의적인 비평가', '초보자' 등 서로 다른 페르소나를 부여합니다.   
* **Step 3\. 협업 기반 다양성 증폭:** 에이전트들은 서로 피드백을 주고받거나 각기 다른 관점에서 텍스트를 생성하여 문체와 어휘의 다양성을 극대화합니다. 

**대안 아키텍처 5: 적응형 대조 학습 (Adaptive Contrastive Learning, ACL)** 모델이 생성한 환각을 부정적 예시로 활용하여 지식 경계를 학습시킵니다. 

* **Step 1\. 지식 사분면 분류:** 모델의 지식 상태를 '아는 것', '불확실한 것', '모르는 것/환각'으로 분류합니다.   
* **Step 2\. 대조 쌍 구성:** 아는 지식은 (질문, 정답)을 긍정 쌍으로, (질문, 환각)을 부정 쌍으로 설정하며, 모르는 지식은 (질문, "모릅니다")를 긍정 쌍으로 설정합니다.   
* **Step 3\. 적응형 손실 함수 최적화:** 생성 손실에 대조 손실을 결합한 적응형 손실 함수를 사용하여 정답 생성 능력과 틀린 정보를 밀어내는 능력을 동시에 기릅니다. 

### **2.4 멀티 에이전트 합의(Multi-Agent Consensus)와 비평**

단일 모델의 편향과 오류를 교차 검증하기 위해 멀티 에이전트 시스템이 도입되고 있다.

* **생성자-비평가(Generator-Critic) 루프:** 하나의 에이전트가 데이터를 생성하면, 다른 에이전트(비평가)가 이를 검토하고 수정 제안을 한다. 이 과정을 통과할 때까지 데이터는 최종 저장소에 커밋되지 않는다.  
* **페르소나 기반 토론:** 서로 다른 관점을 가진 에이전트들이 토론을 통해 합의된 결론을 도출하게 함으로써, 데이터의 중립성과 사실성을 높인다.

### **2.5 자동 프롬프트 최적화 (APO: Automated Prompt Optimization)**

정적인 프롬프트는 정적인 데이터를 낳는다. APO 기술은 프롬프트 자체를 최적화 대상 변수로 취급한다.

* **반복적 개선:** 초기 프롬프트로 생성된 데이터 샘플을 평가하고, 발견된 약점(예: "너무 단조로움", "논리 비약")을 보완하도록 프롬프트를 자동으로 수정한다.  
* **동적 페르소나 주입:** 매 생성 배치를 돌릴 때마다 랜덤한 페르소나, 난이도, 제약 조건을 프롬프트에 동적으로 주입(Injection)하여 데이터셋 전체의 커버리지를 극대화한다.

### **2.6 검색 증강 생성(RAG) 기반 그라운딩(Grounding)**

환각 문제를 해결하기 위해 데이터 생성 시 모델의 내부 지식(Parametric Knowledge)에만 의존해서는 안 된다. 외부 지식 베이스(Vector DB, Knowledge Graph)를 참조하여 생성을 제어해야 한다.

* **지식 그래프(Knowledge Graph) 연동:** 단순 텍스트 검색을 넘어, 지식 그래프 상의 엔티티(Entity) 관계를 기반으로 데이터를 생성함으로써 사실 관계의 논리적 정합성을 보장할 수 있다.  
* **검색기(Retriever) 로직의 일치:** 합성 데이터가 사용될 타겟 애플리케이션의 검색 로직을 데이터 생성 파이프라인에도 동일하게 적용하여, 모델이 "검색된 문맥을 활용하는 법"을 정확히 학습하도록 유도한다.

## **3\. 아키텍처 설계**

위의 모든 리서치 결과와 해결 방안을 통합하여, 엔터프라이즈 환경에서 운영 가능한 **"LLM 합성 데이터 파이프라인"**의 아키텍처를 설계한다. 이 시스템은 LangGraph 와 NVIDIA NeMo 단의 설계 철학을 반영하며, 모듈화, 확장성, 그리고 품질 검증의 자동화에 초점을 맞춘다.

### **3.1 아키텍처 개요도 (Logical Components)**

시스템은 크게 5개의 계층으로 구성된 순환형 그래프(DAG) 구조를 가진다.

* **명세 및 시드 계층 (Specification & Seed Layer) :** 무엇을 만들 것인가?  
* **오케스트레이션 및 생성 계층 (Orchestration & Generation Layer) :** 누가 만들 것인가? (Agentic Core)  
* **검증 및 평가 계층 (Verification & Evaluation Layer) :** 얼마나 잘 만들었는가? (Quality Gate)  
* **정제 및 피드백 계층 (Refinement & Feedback Layer) :** 어떻게 고칠 것인가?  
* **저장 및 운영 계층 (Persistence & Ops Layer) :** 데이터 자산화 및 관리.

### **3.2 계층 1: 명세 및 시드 계층 (Specification & Seed Layer)**

이 계층은 파이프라인의 청사진을 정의하고 초기 엔트로피를 공급한다.

* **스키마 정의 (Schema Definition) :**  
  * **Pydantic/JSON Schema:** 생성될 데이터의 구조를 엄격하게 정의한다. (예: Question, Answer, Context, Reasoning\_Trace 필드 필수). 비정형 텍스트 생성을 지양하고 구조화된 데이터 생성을 강제함으로써 파싱 오류를 원천 차단한다.  
  * **제약 조건:** 필드별 길이 제한, 필수 포함 키워드, 금지어 등을 명시한다.  
* **시드 뱅크 (Seed Bank) :**  
  * **골든 샘플 (Golden Examples):** 인간 전문가가 작성한 고품질 예시 데이터. 퓨샷(Few-shot) 프롬프팅에 사용되어 모델의 스타일을 가이드한다.  
  * **지식 소스 (Knowledge Source):** 도메인 문서(PDF, Wiki), 분류 체계(Taxonomy), 코드베이스 등 RAG에 활용될 원본 데이터.  
* **다양성 샘플러 (Diversity Sampler) :**  
  * 매 생성 요청마다 시드 뱅크에서 서로 다른 조합의 예시와 페르소나를 샘플링하여 프롬프트를 동적으로 구성한다. 이는 데이터셋 전체의 다양성을 확보하는 핵심 메커니즘이다.

### **3.3 계층 2: 오케스트레이션 및 생성 계층 (Generation Layer)**

이 계층은 LangGraph 또는 AutoGen 과 같은 에이전트 프레임워크를 기반으로 상태(State)를 관리하며 데이터를 생성한다.

* **생성 에이전트 (Generator Agent) :**  
  * **역할:** 실제 데이터를 생성하는 LLM 워커.  
  * **기술:** CoT(Chain-of-Thought)를 사용하여 단계별 추론 과정을 포함한 데이터를 생성한다. 오픈라우터(OpenRouter) 등을 통해 라이선스가 확보된 모델(Llama 3, Mistral)을 호출한다.  
* **RAG 그라운딩 엔진 :**  
  * 생성 에이전트가 호출되기 전, 벡터 데이터베이스(Milvus/Weaviate)에서 관련 문맥을 검색하여 프롬프트에 주입한다. 이를 통해 환각을 억제하고 사실에 기반한 생성을 유도한다.  
* **Evol-Instruct 모듈 :**  
  * 단순한 시드 데이터를 받아 복잡도, 추론 단계, 제약 조건을 추가하여 더 어려운 난이도의 데이터로 진화시키는 로직을 수행한다.

### **3.4 계층 3: 검증 및 평가 계층 (Verification & Evaluation Layer) \- "The Judge"**

모델 붕괴를 막는 방파제 역할을 하는 가장 중요한 계층이다. LLM-as-a-Judge 패턴을 적극 활용한다.

* **구문 검증기 (Syntax Validator) :** LLM이 아닌 코드 기반 검증기. Pydantic 스키마를 준수하는지, JSON 형식이 올바른지 확인한다. 실패 시 즉시 재생성을 요청한다.  
* **의미론적 판사 (Semantic Judge) :** GPT-4o 또는 특수 목적의 평가 모델(Prometheus)을 사용하여 생성된 데이터의 품질을 채점한다. 평가 루브릭 (Rubric) : 정확성(Accuracy), 유용성(Helpfulness), 안전성(Safety), 스타일 일치성(Style Consistency) 등 사전에 정의된 기준에 따라 1\~5점 척도로 평가하고, 점수의 근거(Rationale)를 텍스트로 출력한다.  
* **사실성 검증기 (Factuality Checker) :** 생성된 텍스트에서 주장(Claim)을 추출하고, 이를 원본 지식 소스(Knowledge Graph)와 대조하여 지지(Support) 여부를 판단한다.

### **3.5 계층 4: 정제 및 피드백 계층 (Refinement Layer)**

단순히 나쁜 데이터를 버리는 것이 아니라, 이를 수정하여 학습 기회로 삼는다.

* **정제 에이전트 (Refiner Agent) :**  
  * **입력:** 생성된 초안 데이터 \+ 판사(Judge)의 비평 및 점수.  
  * **동작:** 비평 내용을 반영하여 데이터를 수정(Self-Correction)한다. (예: "판사가 설명이 너무 불친절하다고 지적했음. 더 상세한 단계별 설명을 추가하여 다시 작성.")  
* **강화학습(RL) 루프 :** 판사의 평가 결과는 로그로 저장되어, 추후 생성 에이전트의 정책(Policy)을 최적화(PPO, DPO)하거나 보상 모델(Reward Model)을 학습시키는 데 사용된다.

### **3.6 계층 5: 저장 및 운영 계층 (Persistence & Ops Layer)**

* **데이터 레이크하우스:** 최종 승인된 데이터를 Parquet, Iceberg 등의 형식으로 저장한다.  
* **데이터 리니지(Lineage) 추적 :** 모든 데이터 행(Row)에는 메타데이터가 태깅된다: 생성 모델 버전, 프롬프트 템플릿 ID, 참조된 소스 문서 ID, 판사 점수, 생성 일시. 이는 추후 특정 모델이나 프롬프트에 결함이 발견되었을 때, 해당 소스로부터 생성된 데이터만 선별적으로 롤백(Rollback)하거나 제거하는 데 필수적이다.
