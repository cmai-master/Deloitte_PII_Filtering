# **2\. 생성 데이터 정의**

## **2.1  데이터 포맷 구조**

생성되는 합성 데이터는 Text, Tabular, Class 세 컴포넌트로 구성됩니다. 세 컴포넌트는 클래스 내에서 논리적으로 일관성을 유지해야 합니다.

| 컴포넌트 | 내용 | 비고 |
| :---- | :---- | :---- |
| Text data | 마스킹(Asterisk)된 PII 패턴 \+ 앞뒤 \~10자 컨텍스트 | 휴대폰번호·이메일·주민등록번호 3종 |
| Tabular data | Server-i 수집 메타데이터 총 26개 필드(모델에 필요한 feature에 따라 총 필드 수 달라질 수 있음) | 서버6 \+ 파일9 \+ 검출6 \+ 관리5 |
| Class | TP(정탐) 1종 \+ FP(오탐) 7종, 총 8개 클래스 | 균등 분할 생성 (클래스 불균형 방지) |

| ⚠  미결 / 추가 검토 필요 ▸ 마스킹 처리 시점 미결: 생성 전 마스킹(LLM이 마스킹된 형태 직접 생성) vs. 생성 후 마스킹(원본 생성 후 후처리)    → server-i의 마스킹 Rule을 얻을 수 있다면 원본 생성 후 동일하게 마스킹 rule을 적용하는 방식을 채택 (클래스가 올바르게 적용되었는가를 검토하기 쉬움)→ 얻기 불가능 하다면, API 콜 비용을 아끼기 위해 한 프롬프트 내에서 해결하는 방식을 채택 ▸ 클래스-Text-Tabular 삼각 일관성 보장 메커니즘 미설계 (예: ‘FP-개발 더미 데이터’ 클래스라면 Text에 @lguplus 패턴, Tabular 서버그룹도 내부 개발서버류 일치 필요) → 클래스-Text는 일관성을 보장할 순 있지만, 현재 클래스-Tabular는 데이터를 분석해야하기 때문에 현재로서는 보장 불가능 |

# **4\. 생성 방법론 — LLM 기반**

## **4.1  LLM 모델 선택**

사용 LLM은 용도별 모델, 비용 등으로 비교 검토합니다. 만약, 수만건의 데이터를 생성해야한다면 요금 부담을 줄이기 위해 Batch API 사용을 고려할 수 있음 (대량의 요청을 한꺼번에 모아서 처리하는 방식으로 반값으로 24시간 이내 수행).

(LLM 모델은 더 좋은 대체재 발견 시 변경될 수 있음)

| 용도 | LLM 모델 | 비용 | 비고 |
| :---- | :---- | :---- | :---- |
| 생성용 | Claude 4.5 Haiku | 입력 $1.00 / 출력 $5.00 (1M기준) | 비용 부담을 줄이면서도 설계안의 복잡한 스키마를 소화하기에 적합한 모델 |
| 평가용 | GPT-5.4 mini | 입력 $0.75 / 출력 $4.50 (1M기준) | 'Thinking' 모델의 추론 엔진을 경량화하여 탑재한 모델로, 평가 작업에 최적 |

## **4.2  프롬프트 설계 방법론**

**시나리오 기반 생성 전략 (Scenario-based Generation)**

단순히 원본 데이터와 유사한 데이터를 양적으로 늘리는 방식은 학습에 실질적인 효과가 없습니다. 합성 데이터의 핵심 목적은 데이터 분산(Variance)을 넓히는 것이므로, LLM에게 컬럼 정보(필드명, 데이터 타입, 허용 범위)를 제공하여 현실에서 발생 가능한 다양한 시나리오를 먼저 설계하게 한 뒤, 시나리오 단위로 배치 생성을 수행합니다. 이 방식은 동일 프롬프트 반복 호출로 유사 패턴만 쌓이는 Mode Collapse를 구조적으로 방지합니다.

**절차**: ① LLM에게 클래스별 컬럼 정보 제공 → ② LLM이 현실적인 데이터 패턴 및 발생 시나리오 목록 출력 → ③ 시나리오별 배치(N건) 데이터 생성 → ④ 5.1 바이너리 체크리스트로 품질 평가 후 필터링

| ⚠  미결 / 추가 검토 필요 ▸ 클래스별 현실 시나리오 케이스 목록 미작성 → LLM을 활용하여 각 클래스에서 실제로 발생 가능한 시나리오 리스트업 프롬프트 작성 필요 → 클래스별 작성이 필요하며, 공통으로 들어가야할 예시 사항은 아래와 같지만 상황에 따라 일부 수정해야함.① 페르소나 및 상황 설정 (Context): 당신은 기업 내 보안 로그 및 개인정보(PII) 패턴을 완벽히 이해하는 데이터 엔지니어입니다 ② 필드 스키마 정의 (Schema Definition) : Text & Tabular의 필드 명칭, 데이터 타입, 데이터 생성 규칙 등 제공 ③ 한국어 PII 생성 규칙 (Privacy Rules) : 한국적 맥락에 맞는 데이터 생성 규칙 명시 ④ 데이터 다양성 제약 (Variety Constraints) : 모델이 비슷한 패턴만 반복 생성하는 'Mode Collapse'를 막기 위한 지침 (e.g. 동일 IP 주소 3번 이상 반복 금지) ⑤ 출력 형식 지정 (Output Format): 바로 파싱할 수 있도록 “JSON” 형식 ▸ 배치 크기(한 번에 N건) 미산정→ within-batch homogeneity와 API 비용 간 트레이드오프 검토 필요 → 초기 단계에서는 N=50건으로 시작한 후, 중간 TSTR 점검에서 0.75에 가까워지면 N을 150으로 늘리는 방법 채택 |
| :---- |

# **5\. 품질 평가 및 검증**

## **5.1  LLM-as-a-Judge 검증**

생성된 각 레코드를 별도의 LLM이 바이너리(O/X) 체크리스트 방식으로 평가합니다. 점수제(1\~5점)는 점수 척도를 상황마다 다르게 해석할 수 있어 일관성과 신뢰도가 떨어진다는 문제가 있습니다. 구체적인 평가 기준을 항목 단위로 정의하고 각 항목의 통과 여부를 O/X로 판단하여, 통과 항목 수로 데이터 품질을 정량적으로 정의합니다. 생성에 사용한 모델과 동일한 모델을 Judge로 사용하면 자기 출력을 높게 평가하는 편향이 발생하므로, 반드시 서로 다른 LLM으로 교차 검증합니다.

| ⚠  미결 / 추가 검토 필요 ▸ 바이너리 체크리스트 평가 항목 미확정 → 항목 수 및 세부 기준은 확정 후 추가 작성 필요→ (예시) 아래 항목별 O/X 판단 기준을 구체적으로 작성 필요     (1) Text의 PII 마스킹 패턴이 실데이터 Rule을 준수하는가 (O/X)     (2) Tabular 필드값이 해당 클래스의 현실적 시나리오에 부합하는가 (O/X)     (3) 클래스-Text-Tabular 삼각 일관성이 유지되는가 (O/X)     (4) 스키마 필드 누락 또는 타입 오류가 없는가 (O/X)     (5) 완전 일치 기준의 중복 레코드가 아닌가 (O/X) ▸ 폐기 처리 대상을 모델 테스트를 위해 살리는 방향으로 고려 필요 |
