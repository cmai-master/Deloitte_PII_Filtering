# **2\. 생성 데이터 정의**

## **2.1  데이터 포맷 구조**

생성되는 합성 데이터는 Text, Tabular, Class 세 컴포넌트로 구성됩니다. 세 컴포넌트는 클래스 내에서 논리적으로 일관성을 유지해야 합니다.

| 컴포넌트 | 내용 | 비고 |
| :---- | :---- | :---- |
| Text data | 마스킹(Asterisk)된 PII 패턴 \+ 앞뒤 \~10자 컨텍스트 | 휴대폰번호·이메일·주민등록번호 3종 |
| Tabular data | Server-i 수집 메타데이터 총 26개 필드(모델에 필요한 feature에 따라 총 필드 수 달라질 수 있음) | 서버6 \+ 파일9 \+ 검출6 \+ 관리5 |
| Class | TP(정탐) 1종 \+ FP(오탐) 7종, 총 8개 클래스 | 균등 분할 생성 (클래스 불균형 방지) |

፠ 클래스-Text-Tabular 삼각 일관성 보장  
→ 삼각 일관성이란 서로 모순 없이 논리적으로 맞아 떨어지는 상태를 뜻하며, 현재 클래스-Tabular 데이터를 분석할 수 없기 때문에 현재로서는 보장 불가능

## **2.2  Text data 상세**

| PII 유형 | 마스킹 패턴 예시 | 생성 시 유의사항 |
| :---- | :---- | :---- |
| 휴대폰 번호 | 010\*\*\*\*5639 | 한국 번호 형식 (010, 02, 031 등) 준수 등 |
| 이메일 | \*\*\*\*@gmail.com | @lguplus 계열 도메인 고려 등 |
| 주민등록번호 | 110123\*\*\*\*\*\*\* | 뒷자리 첫 숫자 Rule 고려 등 |

፠ 마스킹 데이터를 올바르게 생성하기 위해서는 Server-i의 PII 유형별 마스킹 Rule 확인 필요

**마스킹 처리 시점**

현재 명시적인 PII 마스킹 Rule이 부재해 정규식 기반의 후처리 스크립트 구현이 불가하기 때문에, 프롬프트 내 일괄 마스킹(생성 시점 마스킹) 패턴으로 즉시 생성하는 것으로 채택합니다. 향후 구체적인 마스킹 룰이 확보될 경우, 파이프라인 코드 수정 없이 프롬프트 수정만으로도 유연한 대응이 가능합니다.

## **2.5  데이터 계보(Data Lineage) 및 확장 메타데이터**

모델 학습에 직접 투입되는 핵심 컴포넌트(Class, Text, Tabular)와는 별개로, 생성된 데이터의 출처 추적과 파이프라인 품질 관리를 위해 최종 출력 단계에서 파이프라인 시스템이 별도로 부여하는 메타데이터 컬럼입니다. 이 컬럼들은 LLM이 생성하는 것이 아니라, 생성 파이프라인 스크립트 단에서 자동으로 Wrapping하여 추가합니다. 이 메타데이터는 추후 특정 프롬프트나 모델에서 생성된 데이터에 일괄적인 오류(포맷 붕괴, 특정 패턴 쏠림 등)가 발견되었을 때, 해당 데이터만 조건부로 색출하여 폐기하거나 재생성하는 트러블슈팅 목적으로 활용됩니다.

| 추가 컬럼명 | 설명 | 저장 데이터 예시 | 생성 주체 |
| :---- | :---- | :---- | :---- |
| Is\_Synthetic | 합성 데이터 여부 플래그 | T or Y | 시스템 (파이프라인) |
| Model\_Version | 데이터 생성에 사용된 LLM 모델명 및 버전 | gpt-4o-2024-05-13 | 시스템 (파이프라인) |
| Prompt\_ID | 데이터 생성 지시에 사용된 프롬프트 템플릿의 버전 또는 ID | Prompt\_v1.2\_TP1 | 시스템 (파이프라인) |
| Created\_At | 데이터가 생성된 시점의 타임스탬프 | 2025-03-24 15:30:00 | 시스템 (파이프라인) |
| Batch\_ID | 동일한 API 호출 세션에서 생성된 그룹을 묶는 고유 식별자 | BATCH\_20250324\_TP1\_001 | 시스템 (파이프라인) |

# **3\. 합성 데이터 볼륨 산정**

## **3.1  볼륨 산정 기준**

딥러닝 기반 탐지 모델을 위해 클래스별 최소 1,000건의 합성 데이터 생성을 목표로 확정합니다. 이는 모델의 단순 암기에 의한 과적합을 방지하고, 클래스 내 분산 및 통계적 안정성을 확보하기 위한 정량적 기준입니다. 또한, 학습/검증/평가 데이터 분할 및 향후 RLHF를 통한 고도화 소모량까지 감안할 때, 1,000건의 볼륨은 프로젝트의 신뢰성 보장을 위한 가장 합리적인 임계값으로 산정됩니다.

| 산정 방식 | 기준 | 산출 결과 |
| :---- | :---- | :---- |
| **EPV(Events per Variable) 휴리스틱** | 피처 수(26개) × 10\~20배 (클래스별) | 클래스당 260\~520건 → 전체 2,080\~4,160건 이상 \* 피처 수 변동시, 수정 예정 |
| **딥러닝 휴리스틱** | 클래스별 최소 1,000건 (baseline) | 최소 총 8,000건, 목표 8,000\~80,000건 |
| **TSTR 학습 곡선** | 임계점 도달 후 정확도 향상 미미 | 학습 곡선 도식화 후 적정 볼륨 확정 가능 |

\*휴리스틱: 최적의 해답보다는 '충분히 좋은' 근사치를 찾아내어 노력과 시간을 절약하는 방법

| 📌 참고. Real 데이터 요구량 (합성 데이터 생성을 위한 Seed) • 낮음 (10개 미만 변수): 500\~1,000 레코드 / 프로토타이핑 • 보통 (10\~50개 변수): 5,000\~10,000 레코드 / 표준 ML 분류 ← 현재 26개 피처 해당 • 높음 (50개 이상 변수): 10,000\~25,000+ 레코드 / 고차원 딥러닝 • 시계열: 최소 2\~3년 과거 데이터 (계절성·시간 종속성 보존) |

# **4\. 생성 방법론 — LLM 기반**

## **4.1  LLM 모델 선택**

사용 LLM은 용도별 모델, 비용 등으로 비교 검토합니다. 만약, 수만건의 데이터를 생성해야한다면 요금 부담을 줄이기 위해 Batch API 사용을 고려할 수 있음 (대량의 요청을 한꺼번에 모아서 처리하는 방식으로 반값으로 24시간 이내 수행).

| 용도 | LLM 모델 | 비용 | 비고 |
| :---- | :---- | :---- | :---- |
| 생성용 | Claude 4.5 Haiku | 입력 $1.00 / 출력 $5.00 (1M기준) | 비용 부담을 줄이면서도 설계안의 복잡한 스키마를 소화하기에 적합한 모델 |
| 평가용 | GPT-5.4 mini | 입력 $0.75 / 출력 $4.50 (1M기준) | 'Thinking' 모델의 추론 엔진을 경량화하여 탑재한 모델로, 평가 작업에 최적 |

(LLM 모델은 더 좋은 모델 발견 시 변경될 수 있음)

## **4.2  시나리오 기반 프롬프트 생성 전략**

단순히 원본 데이터와 유사한 데이터를 양적으로 늘리는 방식은 학습에 실질적인 효과가 없습니다. 합성 데이터의 핵심 목적은 데이터 분산(Variance)을 넓히는 것이므로, LLM에게 컬럼 정보(필드명, 데이터 타입, 허용 범위)를 제공하여 현실에서 발생 가능한 다양한 시나리오를 먼저 설계하게 한 뒤, 시나리오 단위로 배치 생성을 수행합니다. 이 방식은 동일 프롬프트 반복 호출로 유사 패턴만 쌓이는 Mode Collapse를 구조적으로 방지합니다.

**프롬프트 생성 절차**

① LLM에게 제공할 컬럼 메타데이터 정보 제공  
→ 생성해야 하는 모든 속성(Text \+ Tabular 데이터 필드)의 정의와 제약 조건을 명시한 가이드라인 작성

② LLM이 ‘시나리오 기획’ 프롬프팅  
→ LLM의 추론 능력을 활용하여 클래스별(정탐/오탐)로 발생 가능한 시나리오를 도출  
→ 프롬프트 예시: 해당 클래스에서 발생할 수 있는 현실적인 상황 5가지를 육하원칙에 맞춰 기획하라  
③ 시나리오 기반의 다차원 Variance 확장  
→ 2단계에서 나온 시나리오의 핵심 뼈대는 유지하되, 속성값을 여러가지로 변경해 파생 케이스 생성

→ 프롬프트 예시: 상황 A를 바탕으로 규모를 키운 케이스(대용량 파일, 긴 실행 시간), 행위자를 바꾼 케이스(배치 계정이 아닌 일반 개발자 계정), 경로를 바꾼 케이스(리눅스 경로가 아닌 윈도우 경로)로 분산을 줘서 시나리오를 3배로 확장하라

④ 시나리오별 배치 데이터 생성

→ JSON 형태로 설정된 배치 크기에 맞게 데이터 생성

⑤ 5.1 바이너리 체크리스트로 품질 평가 후 필터링

**배치 크기 산정 및 운영 방안**

LLM의 컨텍스트 윈도우 한계와 API 호출 비용을 고려하여, 데이터 생성 배치를 2단계로 나누어 운영합니다. 

| 운영 단계 | 배치 크기 | 검증 목적 및 기대 효과 | 세부 조건 및 고려 사항 |
| :---- | :---- | :---- | :---- |
| Phase 1: 초기 생성 및 검증 | 50건 | 프롬프트 지침 준수 여부, 삼각 일관성, 마스킹 규칙 적용 상태 확인 | 1회 API 호출 시 50건 레코드 생성 및 TSTR 중간(클래스별 300건 도달 시) 점검 수행 |
| Phase 2: 본 생성 및 볼륨 확대 | 150건 | 타겟 볼륨을 비용 효율적이고 신속하게 달성 | Phase1의 TSTR 점수가 임계값(0.75)을 통과했을 경우에만 새로운 배치 크기를  적용하며, JSON 포맷 붕괴나 지침 누락을 지속 모니터링 |

# **5\. 품질 평가 및 검증**

## **5.1  LLM-as-a-Judge 검증**

생성된 각 레코드를 별도의 LLM이 바이너리(O/X) 체크리스트 방식으로 평가합니다. 점수제(1\~5점)는 점수 척도를 상황마다 다르게 해석할 수 있어 일관성과 신뢰도가 떨어진다는 문제가 있습니다. 구체적인 평가 기준을 항목 단위로 정의하고 각 항목의 통과 여부를 O/X로 판단하여, 통과 항목 수로 데이터 품질을 정량적으로 정의합니다. 생성에 사용한 모델과 동일한 모델을 Judge로 사용하면 자기 출력을 높게 평가하는 편향이 발생하므로, 반드시 서로 다른 LLM으로 교차 검증합니다.

**바이너리 체크리스트 세부 평가 항목 및 판단 기준**

생성된 레코드는 별도의 LLM(Judge) 및 Rule-based 스크립트를 통해 아래 7가지 항목에 대해 바이너리(O/X) 방식으로 교차 검증을 받으며, 이를 모두 통과해야 최종 데이터셋으로 편입됩니다. 

| 평가 항목 | 적용 여부 | O 판단 기준 | X 판단 기준 | 관련 컴포넌트 |
| :---- | :---- | :---- | :---- | :---- |
| **(1) PII 마스킹 패턴 실데이터 Rule 준수 여부** | 적용 | 휴대폰 번호(한국 번호 형식), 이메일 등 각 PII 유형별 마스킹(Asterisk) 패턴 및 앞뒤 컨텍스트가 규칙에 맞게 생성됨. | 마스킹이 누락되어 평문으로 노출되거나, 지정된 한국적 맥락 및 도메인 규칙을 벗어난 패턴이 생성됨. | Text |
| **(2) Tabular 필드값의 현실적 시나리오 부합 여부** | 적용 | 26개 필드의 값들이 해당 클래스(정탐/오탐 8종)에서 실제로 발생 가능한 시나리오 문맥에 논리적으로 부합함. | 특정 클래스의 맥락과 무관한 의미 없는 더미 데이터가 들어가거나, 필드 간의 논리적 연결성이 떨어짐. | Tabular |
| **(3) 클래스-Text-Tabular 삼각 일관성 유지 여부** | 보류 | 세 컴포넌트가 논리적으로 일치함 (예: 'FP-내부망' 클래스 시 Text에 내부 시스템 패턴, Tabular의 IP 대역도 내부망으로 일치). | 클래스 레이블의 정의와 Text의 패턴, 혹은 Tabular의 메타데이터 간에 모순이나 불일치가 발생함. | Class, Text, Tabular |
| **(4) 스키마 필드 누락 및 타입 오류 여부** | 적용 | 지정된 JSON 스키마에 따라 필요한 필드가 모두 존재하며, 요구되는 데이터 타입(문자열, 숫자, 불리언 등)을 정확히 준수함. | 필수 필드가 누락되었거나, 숫자형 필드에 문자열이 들어가는 등 데이터 타입 오류가 발생함. | 전체 |
| **(5) 완전 일치 기준 중복 레코드 여부** | 적용 | 기존에 생성되었거나 학습된 데이터와 '완전 일치'하는 레코드가 아님. | 기존 DB에 존재하는 레코드와 모든 필드 값이 완전히 동일하게 중복됨. | 전체 |
| **(6) 프롬프트 세부 지침 엄격 준수 여부** | 적용 | JSON 포맷 외의 불필요한 서술어("네, 생성해 드립니다" 등)가 없으며, 프롬프트에서 요구한 특수 조건(예: 파일 확장자 제약, 길이 제한)을 모두 충족함. | 지정된 JSON 밖의 텍스트가 섞여 있어 Parser 오류를 유발하거나, 명시된 데이터 생성 제약사항을 누락함. | 전체 (Format) |
| **(7) 배치내 패턴 다양성 및 의미적 중복 방지** | 적용 | 동일 생성 배치 내에서 특정 이름, 회사명, IP 대역 등의 핵심 패턴이 과도하게 반복되지 않고 다양성을 확보함. | 완전 일치는 아니더라도, 템플릿의 조사만 바꾼 수준의 의미적 중복이 발생하거나 특정 값으로의 편향(Mode Collapse)이 발생함. | 전체 (Batch) |

## **5.2  \[Optional\] 폐기 처리 데이터 재활용 및 모델 테스트 활용 방안**

바이너리 체크리스트(포맷 오류, 스키마 불일치, 마스킹 누락 등)를 통과하지 못해 1차적으로 폐기 판정을 받은 합성 데이터를 즉시 삭제하지 않고, 탐지 모델의 견고성 및 예외 처리 능력을 검증하기 위한 테스트 데이터셋으로 전환하여 재활용합니다.

**세부 활용 시나리오**

| 시나리오명 | 적용 데이터 특징 | 기대 효과 및 검증 목적 |
| :---- | :---- | :---- |
| **Hard Negative / Edge Case 테스트 셋 구축** | 의도된 생성 규칙을 벗어난 불완전 패턴 모호한 문맥이나 정제되지 않은 노이즈 데이터 | 모델이 이를 과탐(False Positive)하지 않는지 검증 예외 데이터를 정확히 걸러내거나 보류 처리하는 능력 평가 |
| **시스템 스트레스 테스트 (예외 처리 검증)** | 스키마 필드가 누락된 데이터 타입 오류 등 포맷이 망가진 Tabular/Text 데이터 | 추론 파이프라인(분류 모델, 파서 등)의 예외 처리 로직 작동 여부 확인 시스템 장애 없이 방어 및 정상 로깅이 이루어지는지 검증 |
| **프롬프트 엔지니어링 피드백 루프** | 특정 클래스나 프롬프트 지침에서 평가 탈락된 데이터 | 생성 실패율이 높은 원인 역추적 및 분석 실패 케이스를 통한 데이터 생성 프롬프트 취약점 파악 및 고도화 |

**파이프라인 처리 및 저장 로직**

1) 태깅 및 분리 적재  
   : 체크리스트 평가에서 탈락한 레코드는 메인 학습/평가 파이프라인에서 분리하되, 각 데이터에 Discard\_Reason(예: 스키마 위반, 마스킹 오류, 중복 등) 및 Prompt\_ID 메타데이터를 태깅  
2) 별도 DB 운용  
   : 정탐/오탐 클래스와 섞이지 않도록 Rejected\_Pool (또는 Edge\_Case\_DB) 등의 별도 스토리지에 격리 보관하며, 추후 필요시 조건별로 쿼리하여 테스트 스크립트에 연동할 수 있도록 구성

## **5.3  평가 지표**

| 지표 | 설명 | 현재 상태 |
| :---- | :---- | :---- |
| **충실도 (Fidelity)** | 원본 데이터의 다차원 통계 구조 재현 여부 (KL Divergence, 상관관계 행렬 등) | 원본 데이터 부족 → 당장 적용 불가 (추후 실데이터 확보 시 검증 가능) |
| **유용성 (Utility)** | TSTR(Train on Synthetic, Test on Real) 성능 비교 | 핵심 KPI — TSTR F1(임계값: 0.75) 기준 재생성 트리거 연동 |
| **분류 모델 KPI** | 합성 데이터로 학습한 분류 모델의 TP/FP 구분 정확도 | 최종 파이프라인 성공 기준으로 정의 필요 |
